05 WhatsApp conversation state machine
State is stored in Redis as a hash keyed by conv:{owner_phone}. Expires after 4 hours of inactivity. Every incoming message first reads state, then decides how to handle the message.
IDLE
No active conversation. Waiting for a new inquiry to be forwarded.
message received → PARSING
onboarding trigger → ONBOARDING
ONBOARDING
First-time setup. Collecting business name, GST, catalog, payment terms. Step tracked in state.
step complete → ONBOARDING (next step)
all steps done → IDLE
PARSING
Inquiry received, Claude is parsing it. Owner gets a "processing" acknowledgment. Short-lived.
parse success → AWAITING_PRICE
parse failed → AWAITING_CLARIFICATION
AWAITING_PRICE
Summary sent to owner. Waiting for them to reply with a price. inquiry_id stored in state.
valid price reply → AWAITING_APPROVAL
"skip" → IDLE
4hr timeout → IDLE (flagged)
AWAITING_CLARIFICATION
Parser couldn't extract a field with confidence. One clarifying question sent to owner.
owner replies → PARSING (retry)
"skip" → IDLE
AWAITING_APPROVAL
Draft proposal shown to owner. Waiting for SEND, EDIT, or SKIP command.
"SEND" → IDLE (PDF sent)
"EDIT" → AWAITING_PRICE
"SKIP" → IDLE
# services/conversation.py — state read/write import json from app.db.redis import redis_client STATE_TTL = 4 * 60 * 60 # 4 hours async def get_state(owner_phone: str) -> dict: raw = await redis_client.get(f"conv:{owner_phone}") return json.loads(raw) if raw else {"state": "IDLE"} async def set_state(owner_phone: str, data: dict): await redis_client.set( f"conv:{owner_phone}", json.dumps(data), ex=STATE_TTL ) async def clear_state(owner_phone: str): await redis_client.delete(f"conv:{owner_phone}") # Usage in webhook handler async def handle_whatsapp_message(phone: str, message: str): state = await get_state(phone) if state["state"] == "AWAITING_PRICE": await handle_price_reply(phone, message, state) elif state["state"] == "AWAITING_APPROVAL": await handle_approval(phone, message, state) else: # New forwarded inquiry — enqueue parse task parse_inquiry.delay(phone, message) await set_state(phone, {"state": "PARSING"})
06 Inquiry parser — Claude prompt design
# services/parser.py SYSTEM_PROMPT = """ You are an inquiry parser for Indian B2B manufacturing businesses. Extract structured data from buyer inquiries. Inquiries may be in English, Hindi, Hinglish, or mixed. Be liberal in interpretation. Return ONLY valid JSON. No preamble. No explanation. Schema: { "product_name": "string — normalized product name", "quantity": "number — numeric value only", "unit": "string — kg / MT / piece / roll / box / meter", "specs": "object — grade, size, thickness, etc. as key-value", "delivery_location": "string or null", "delivery_deadline": "string or null — ISO date if determinable", "urgency": "low | medium | high", "buyer_name": "string or null", "confidence": "number — 0.0 to 1.0", "missing_fields": ["list of fields that could not be extracted"] } """ async def parse_inquiry(raw_text: str, catalog: list[dict]) -> dict: catalog_context = json.dumps(catalog, ensure_ascii=False) response = await anthropic_client.messages.create( model="claude-sonnet-4-6", max_tokens=1000, system=SYSTEM_PROMPT, messages=[{ "role": "user", "content": f""" Business catalog: {catalog_context} Inquiry to parse: {raw_text} """ }] ) result = json.loads(response.content[0].text) # If confidence below threshold, flag for clarification if result["confidence"] < 0.75 or result["missing_fields"]: result["needs_clarification"] = True return result

08 Celery tasks — async + scheduled
# tasks/follow_up.py from celery import shared_task from datetime import timedelta from app.tasks.celery_app import celery_app def schedule_follow_ups(proposal_id: str, sent_at: datetime): # Schedule 3 follow-up emails after proposal send delays = [2, 5, 10] # days for day in delays: eta = sent_at + timedelta(days=day) task = send_follow_up_email.apply_async( args=[proposal_id, day], eta=eta ) # Store task ID for cancellation FollowUpJob.create( proposal_id=proposal_id, send_at=eta, celery_task_id=task.id, status="pending" ) @celery_app.task def send_follow_up_email(proposal_id: str, day: int): proposal = Proposal.get(proposal_id) job = FollowUpJob.get(proposal_id=proposal_id, day=day) # Check if cancelled (buyer replied) if job.status == "cancelled": return subject, body = build_followup_copy(proposal, day) email_sender.send( to=proposal.inquiry.buyer_email, from_=proposal.business.owner_email, subject=subject, body=body, reply_to=proposal.business.owner_email ) job.update(status="sent") def cancel_follow_ups(proposal_id: str): # Called when buyer replies to any email jobs = FollowUpJob.get_pending(proposal_id) for job in jobs: celery_app.control.revoke(job.celery_task_id) job.update(status="cancelled")
09 Hosting recommendation
Recommended for MVP
Railway
~$20–40/mo
Deploy from GitHub in minutes
FastAPI + Celery worker as separate services
Managed PostgreSQL + Redis built in
Automatic SSL, no Nginx config
Zero DevOps to start
Pricier at scale vs AWS
Less control over infra
Render
~$25–50/mo
Similar to Railway, slightly more mature
Background workers supported
Good free tier for early testing
Celery worker setup slightly more manual
Cold starts on free tier
AWS (EC2 / ECS)
~$30–80/mo
Full control, cheapest at scale
RDS PostgreSQL + ElastiCache Redis
Significant setup time (VPC, SGs, IAM)
Wrong tradeoff for 10-day MVP
Save this for post-product-market-fit
Use Railway for MVP. Add Cloudflare in front for webhook reliability and DDoS protection — 360dialog will retry webhooks on any non-200 response, so your endpoint must be fast and stable. Cloudflare's free tier handles this.

11 Recommended build order
Build in dependency order. Each day produces something testable. Never build two things at once that depend on each other.
Day 1 — Foundation
FastAPI app skeleton + config.py with pydantic-settings
PostgreSQL connection + SQLAlchemy models + first Alembic migration
Redis connection + basic get/set helpers
Deploy skeleton to Railway — confirm /health returns 200
Day 2 — Channels
360dialog webhook endpoint — verify signature, echo bot response
Phone-number-to-business routing
Gmail API polling task — store raw emails in DB
Day 3 — Parser
Claude inquiry parser with structured JSON output
Test against 20 real samples from manual sprint
Whisper voice-to-text for audio messages
Day 4 — State machine
Redis conversation state — all 6 states wired up
Celery task queue — parse_inquiry task running async
Owner summary WhatsApp message sent after parse
Day 5 — Onboarding flow
ONBOARDING state — step-by-step WhatsApp conversation
Business + catalog_items written to DB from conversation
Logo upload flow via WhatsApp image → R2
Day 6 — Price + approval flow
Price reply handling — AWAITING_PRICE → AWAITING_APPROVAL
Draft proposal preview message to owner
SEND / EDIT / SKIP command handling
Day 7 — PDF generator
proposal.html Jinja2 template — professional layout
WeasyPrint PDF generation + R2 upload
Email delivery with PDF attachment via SendGrid
Day 8 — Follow-up scheduler
Celery beat follow-up tasks — day 2, 5, 10
Cancellation on buyer reply — revoke pending tasks
Owner WA notification when buyer replies
Days 9–10 — Integration + testing
Full end-to-end test with real business data from manual sprint
Edge cases: missing buyer email, voice note inquiries, EDIT flow
First beta user onboarded — live on production
